## 1. Executive Summary

### 1.1 The Distribution Problem

The Wheeler ecosystem has built eight revenue systems across three hardened nodes with 61 services, 41 Docker containers, and a 100/100 QA score. The infrastructure is ready. The products exist. But zero customers can find them.

No organic search presence. No content marketing. No developer ecosystem. No referral infrastructure. No self-service acquisition funnel. The attorney marketplace exists as a manual spreadsheet of four names. SurplusAI has no signup flow. Prediction Radar has Stripe in test mode with no subscriber management.

Distribution is the critical missing layer between infrastructure readiness and revenue production.

This document defines eight integrated growth engines that work together to drive qualified traffic, convert users, and scale revenue across the Wheeler ecosystem:

### 1.3 Guiding Principles

1. **Measure everything.** Every channel has specific metrics, targets, and tracking infrastructure. No channel operates without measurement.

2. **No fake projections.** All keyword estimates, conversion rates, and revenue projections are based on real legal/surplus fund industry benchmarks, not aspirational math.

3. **Infrastructure-first.** Every growth channel maps to existing infrastructure (Usesend :3007 for email, predictionradar.ai for SaaS, surplusai.io for PLG) before building new tooling.

4. **Compound growth.** Each engine feeds the others: content fuels SEO, SEO drives signups, signups generate referrals, referrals create content.

5. **Freemium economics work.** SurplusAI Basic (1 county, 50 leads/month) costs approximately $4.17/month in infrastructure at current scale -- viable at $0 with a 3-5% Pro conversion target.

#### Template Generation Architecture

```bash
# /opt/wheeler/scripts/generate-county-pages.sh
# Runs weekly via cron, processes all 3,000+ US counties

INPUT=/opt/wheeler/data/counties/us-counties.csv
TEMPLATES=/opt/wheeler/www/surplusai.io/templates/
OUTPUT=/opt/wheeler/www/surplusai.io/county/

while IFS=, read -r county state fips; do
  for type in guide attorneys data; do
    node /opt/wheeler/scripts/render-template.js \
      --template "${TEMPLATES}/county-${type}.html" \
      --county "$county" --state "$state" --fips "$fips" \
      --output "${OUTPUT}${state,,}/${fips}/${type}/index.html"
  done
done < "$INPUT"

# Post-generation: sitemap update
node /opt/wheeler/scripts/generate-sitemap.js --base https://surplusai.io --pages $OUTPUT
```

#### SEO Considerations for Programmatic Pages

- **No duplicate content:** Each page type has 40%+ unique content via county-specific court info, demographic data, and state legal variations. The common framework is identical but the substantive content is unique.
- **Interlinking:** Each page links to related pages: county guide -> county attorney page -> state guide -> blog posts about that state's laws.
- **Indexing priority:** Submit via sitemap in order of county population (Los Angeles first, Loving County Texas last).
- **Index coverage target:** 80%+ of generated pages indexed within 60 days. Pages with thin content (<300 words of unique text) get noindex.

#### Crawl Budget Management

At 9,000+ pages, crawl budget management is critical:

1. **Sitemap prioritization:** Top 500 counties (by population) in sitemap priority 1.0. Remaining counties at priority 0.5. Updated weekly as content changes.
2. **Internal linking depth:** All county pages reachable within 3 clicks from homepage: Home -> [State] -> [County]. Breadcrumb navigation on every page.
3. **Orphan page prevention:** Every county page links to its state page, which links back to the top 10 county pages. Remaining county links in a paginated list (100 per page).
4. **Thin content detection:** Weekly script runs content-length analysis. Pages below 300 unique-content words get `noindex` until enriched.

## 5. Referral Systems

### 5.1 Attorney Referral Program

**Structure:** Revenue share on referred attorneys' first 6 months of marketplace fees.

**Mechanism:**
- Existing attorney gets a unique referral link: `attorney.fundsrecoverygroup.com/ref/{code}`
- Referred attorney signs up via link (tracked via cookie + referral code in signup form)
- Referred attorney completes onboarding, gets verified, starts receiving case assignments
- Referring attorney receives 10% of FRG's revenue share from referred attorney's cases for 6 months
- Cap: $2,000 per referred attorney

**Economics:**
- Average FRG revenue per attorney case: $375 (25% of $1,500 average surplus recovery fee)
- Average attorney cases/month: 5 (after ramp-up, 3 months)
- Referring attorney earn: $375 x 10% x 5 cases x 6 months = $112.50 per referred attorney
- Breakeven for FRG: referred attorney generates >6 months of fees to cover referral cost
- CAC for referral-sourced attorneys: $112.50 (vs. estimated $500+ for paid acquisition)

**Tracking:**
- Referral code stored in `marketplace.attorneys.referral_code` and `referred_by`
- Automatically applied during attorney registration
- Payouts calculated monthly via revenue share engine (`attorney-revenue-engine` PM2 :8124)
- Referral status dashboard in attorney portal

### 5.2 Client Referral Program

**Structure:** Finder's fee for successful claimant referrals (individuals who refer friends/family with surplus fund cases).

**Mechanism:**
- Claimant (or any individual) submits a referral via `surplusai.io/refer` form
- Referral includes: claimant name, property address, county, contact info
- FRG intakes the referral, validates surplus fund existence, initiates claim process
- If claim results in recovery, referee receives 5% of FRG's fee (not the claimant's recovery)
- Maximum payout: $500 per successful referral

**Economics:**
- Average FRG fee per case: $375
- Referee payout: $18.75 (5% of $375) to maximum $500
- This is intentionally modest to avoid regulatory issues with fee splitting for legal referrals
- Key differentiator: referee is not an attorney and is not referring legal services -- they are referring a claim opportunity that FRG evaluates independently

**Regulatory consideration:** Most states prohibit non-attorneys from receiving referral fees for legal services. This program is structured as a finder's fee for introducing a business opportunity (the unclaimed surplus fund), not for referring legal representation. The referee is compensated when FRG successfully recovers funds, which is a contingent business transaction, not a legal fee split. Legal review required before launch.

### 8.3 Key Metric Definitions

#### Customer Acquisition Cost (CAC) by Channel

```
CAC_channel = Total channel spend (month) / New customers from channel (month)

Channel spend includes:
  - SEO: content creation cost ($/article x articles/month) + tool subscriptions (Ahrefs, etc.)
  - Content: content creation + distribution costs
  - Referral: referral payouts + program management
  - Partner: partner commissions + partnership management
  - PLG: product development cost allocated to growth features
  - Paid: ad spend (if applicable, not in current plan)

Blended CAC = Total all channel spend / Total new customers
```

#### Lifetime Value (LTV) by Segment

```
LTV_segment = ARPU_segment / Monthly churn rate_segment

Where:
  - ARPU (Basic) = $0 (but track engagement as proxy for referral value)
  - ARPU (Pro) = Average Pro subscription revenue per customer
  - ARPU (Enterprise) = Average Enterprise subscription revenue per customer
  - ARPU (API) = Average API usage revenue per developer

LTV:CAC ratio target: > 3:1 for each segment
```

#### Viral Coefficient (k-factor)

```
k = i x c

Where:
  i = invitations sent per existing user (per period)
  c = conversion rate of invitations to new active users

Measured per referral loop (attorney sharing, workspace invites, claimant sharing)
Target combined k-factor: > 0.5 by month 6
```

#### Net Revenue Retention (NRR)

```
NRR = (Starting MRR + Expansion MRR - Churn MRR) / Starting MRR

Where:
  - Expansion MRR: upgrades, additional seats, API usage increases
  - Churn MRR: downgrades, cancellations, non-payment

Target NRR: > 100% (expansion exceeds churn)
```

**Rate limiting by signup step:**

| Step | Limit | Window | Rationale |

### 9.2 Guided County Selection Wizard

The county selection wizard is the most critical step in onboarding -- it determines the user's first experience and directly impacts activation.

**Wizard flow:**

```
Step 5a: Location Detection
  - IP geolocation: "We see you're in [City], [State]. Is this where you practice?"
  - If yes: pre-select user's home county
  - If no: show search with state dropdown

Step 5b: County Search and Discovery
  - Search bar: type county name, state
  - Browse by state: expandable list of counties per state
  - Map view: click county on interactive US map
  - Recommendations: "Top 10 counties by surplus fund volume"

Step 5c: County Details (on hover/select)
  - County name, state
  - Active cases in last 30 days
  - Average surplus amount
  - Total estimated unclaimed surplus
  - Number of attorneys currently serving this county
  - "Estimated recovery potential: $X based on county average"

Step 5d: Selection Confirmation (Basic only)
  - "You selected [County], [State]"
  - "We'll monitor this county for new surplus fund cases and email you weekly"
  - "Tip: Upgrade to Pro to add up to 5 counties"

Step 5e (Pro/Trial): Multi-County Selection
  - Add counties to a "monitoring list"
  - Drag to reorder priority
  - See total estimated recovery potential across all selected counties
  - "You're monitoring $X in estimated surplus funds across [N] counties"
```

**Engineering:** The interactive county map uses Mapbox GL JS with a county GeoJSON boundary dataset (compressed to ~3MB for all 3,000+ counties). County data (case counts, averages) is served from a Redis cache populated by a daily cron job querying the surplus fund database.

### 11.4 Revenue Attribution Model

Every lead, signup, and conversion is attributed to a source channel using a combination of:

1. **UTM parameters** on all marketing outbound links (source, medium, campaign, content)
2. **Referral codes** for all referral/affiliate/partner programs
3. **First-touch attribution** for new signups (channel that drove the first visit)
4. **Last-touch attribution** for conversions (channel that drove the conversion visit)
5. **Multi-touch attribution** (linear model) for reporting dashboards

```python
# Attribution logic pseudocode

def attribute_conversion(user_id, conversion_type):
    """
    Assign revenue credit to channels based on touchpoints.
    
    Returns array of {channel, percentage, revenue_credit}
    """
    touchpoints = get_user_touchpoints(user_id, conversion_type)
    
    if conversion_type == 'signup':
        # First-touch for signups
        return [{touchpoints[0].channel: 1.0}]
    
    elif conversion_type == 'trial_start':
        # Last-touch for trial starts
        return [{touchpoints[-1].channel: 1.0}]
    
    elif conversion_type == 'subscription':
        # Multi-touch (linear) for paid conversions
        weight = 1.0 / len(touchpoints)
        return [{tp.channel: weight for tp in touchpoints}]
    
    elif conversion_type == 'referral':
        # Referral code attribution (overrides UTM)
        referral = get_referral_code(user_id)
        return [{referral.channel: 1.0}]
```

### A.3 Generation Script

```bash
#!/bin/bash
# /opt/wheeler/scripts/generate-county-pages.sh
# Runs weekly via cron, processes all 3,146 US counties

set -euo pipefail

DATA_DIR="/opt/wheeler/data/counties"
TEMPLATE_DIR="/opt/wheeler/www/surplusai.io/templates"
OUTPUT_DIR="/opt/wheeler/www/surplusai.io/county"
LOG_FILE="/var/log/wheeler/county-pages-$(date +%Y%m%d).log"

mkdir -p "$OUTPUT_DIR"
exec > "$LOG_FILE" 2>&1

echo "[$(date)] Starting county page generation..."
echo "[$(date)] Processing $(wc -l < "$DATA_DIR/us-counties.csv") counties"

while IFS=, read -r county state fips pop court_name court_addr; do
  state_lower=$(echo "$state" | tr '[:upper:]' '[:lower:]')
  county_dir="$OUTPUT_DIR/$state_lower/$fips"

  for type in guide attorneys data; do
    mkdir -p "$county_dir/$type"
    node /opt/wheeler/scripts/render-template.js \
      --template "$TEMPLATE_DIR/county-${type}.html" \
      --data "$DATA_DIR/county-data.json" \
      --fips "$fips" \
      --output "$county_dir/$type/index.html"
  done
done < "$DATA_DIR/us-counties.csv"

echo "[$(date)] Generating sitemap..."
node /opt/wheeler/scripts/generate-sitemap.js \
  --base "https://surplusai.io" \
  --pages "$OUTPUT_DIR" \
  --output "/opt/wheeler/www/surplusai.io/sitemap-counties.xml"

echo "[$(date)] Pinging Google..."
curl -s "https://www.google.com/ping?sitemap=https://surplusai.io/sitemap-counties.xml" > /dev/null

echo "[$(date)] County page generation complete. Total pages: $(find "$OUTPUT_DIR" -name 'index.html' | wc -l)"
```

## Appendix B: UTM Parameter Framework

### B.1 Standard Parameters

Every outbound marketing link follows this convention:

```
?utm_source={source}&utm_medium={medium}&utm_campaign={campaign}&utm_content={content}&utm_term={term}
```

### B.2 Parameter Values

| Parameter | Values | Used By |
|-----------|--------|---------|
| `utm_source` | `google`, `linkedin`, `twitter`, `youtube`, `facebook`, `reddit`, `newsletter`, `referral`, `partner`, `affiliate`, `blog`, `direct` | All channels |
| `utm_medium` | `organic` (SEO), `social` (social media), `email` (newsletter), `cpc` (if paid), `referral` (referral program), `affiliate` (affiliate), `partner` (partner), `blog` (internal blog), `video` (YouTube) | All channels |
| `utm_campaign` | `surplusai-launch`, `frg-referral-program`, `pro-trial-q2-2026`, `surplus-intelligence-weekly`, `bar-partnership-[state]`, `county-seo-launch` | Campaign-specific |
| `utm_content` | `hero-cta`, `sidebar`, `inline-link`, `footer-cta`, `banner`, `popup`, `email-cta`, `video-description`, `bio-link` | Content-specific |
| `utm_term` | SEO keyword target (e.g., `surplus+funds+california`) | SEO (Google Ads if applicable) |

### B.3 Referral Code Parameter

For referral/affiliate/partner links, replace `utm_source=referral` with:

```
?ref={referral_code}
```

The referral code is automatically linked to the referrer's account during signup registration.
